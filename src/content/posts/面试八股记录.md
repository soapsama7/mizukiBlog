## Redis

### Redis 为什么快？

Redis 性能高主要有四个核心原因。

第一，Redis 基于内存进行数据读写，不依赖磁盘 IO，而内存访问速度远高于磁盘，这是 Redis 高性能最根本的原因。

第二，Redis 底层使用了大量高效的数据结构，例如 HashTable、SkipList、QuickList、SDS 等，针对不同场景做了专门优化，提高了读写效率。

第三，Redis 的核心命令执行采用单线程模型，避免了多线程环境下的锁竞争、线程切换和上下文切换开销，在高频读写场景下反而更高效。

第四，Redis 使用 IO 多路复用机制（如 epoll），可以用一个线程同时处理大量客户端连接，而不是一个连接对应一个线程，大幅提升了高并发处理能力。

因此 Redis 的高性能并不仅仅是因为“基于内存”，而是内存、高效数据结构、单线程模型和 IO 多路复用共同作用的结果。



## MySQL

### 什么情况下会导致 MySQL 索引失效

MySQL 索引失效常见场景主要包括：

第一，联合索引没有遵循最左匹配原则，例如索引是 `(a,b,c)`，查询条件只使用了 `b`，没有使用最左列 `a`。

第二，对索引列使用函数或表达式运算，例如 `year(create_time)` 或 `age + 1`，会破坏索引结构，导致无法使用索引。

第三，发生隐式类型转换，例如字符串字段用数字去查询，可能导致索引失效。

第四，使用 `%` 开头的模糊查询，例如 `like '%abc'`，由于无法确定起始位置，索引通常无法生效。

第五，使用 `or` 查询时，如果其中一侧没有索引，优化器可能直接选择全表扫描。

另外，MySQL 优化器本身也会根据成本判断是否使用索引。如果表数据量较小、命中数据过多或者回表成本较高，即使存在索引，也可能主动放弃索引而选择全表扫描。



### MySQL为啥选B+Tree

MySQL 选择 B+Tree 而不是红黑树，核心原因是数据库查询的瓶颈主要在磁盘 IO，而不是内存计算。

红黑树本质上是二叉平衡树，每个节点最多只有两个子节点，当数据量较大时，树的高度会明显增加。

树高越高，意味着一次查询需要访问更多磁盘页，也就需要更多磁盘 IO，查询性能会明显下降。

而 B+Tree 属于多路平衡查找树，一个节点可以存放大量 key，并拥有更多子节点，因此树的高度非常低。

即使在千万级数据量下，通常只需要 3～4 层就可以完成查询，大幅减少了磁盘 IO 次数。

因此在数据库这种以磁盘访问为主要成本的场景下，B+Tree 比红黑树更适合作为索引结构。

MySQL 使用 B+Tree 而不是 B树，主要有三个原因。

第一，B+Tree 的非叶子节点只存储索引 key，不存储实际数据，因此同样大小的磁盘页可以存放更多 key，单个节点的分叉数量更多，整棵树的高度更低，从而减少磁盘 IO 次数。

第二，B+Tree 的所有数据都存储在叶子节点，并且叶子节点之间通过双向链表连接，这使得范围查询非常高效。比如 between、order by、范围扫描等操作，都可以直接顺着叶子节点链表遍历。

而 B树没有这种天然的链表结构，范围查询效率较差。

第三，B+Tree 所有查询都必须走到叶子节点，查询路径长度稳定，性能更可预测；而 B树可能在非叶子节点就命中，查询路径不固定。

因此在数据库这种大量范围查询、强调磁盘 IO 优化的场景下，B+Tree 比 B树更适合作为索引结构。



### 事物四大特性

* 原子性（Atomiccity)：一个事务中的所有操作，要么全部完成，要么全部不完成，不会结束在中间某个环节，而且事务在执行过程中发生错误，会被回滚到事务开始前的状态，就像这个事务从来没有执行过一样，就好比买一件商品，购买成功时，则给商家付了钱，商品到手；购买失败时，则商品在商家手中，消费者的钱也没花出去。
* 一致性（Consistency）：是指事务操作前和操作后，数据满足完整性约束，数据库保持一致性状态。比如，用户A和用户B在银行分别有800元和600元，共总共1400元，用户A给用户B转账200元，分为两个步骤，从A的账户扣除200元和对B的账户增加2200元。一致性就是是要求上述步骤操作后，最后的结果是用户A还有600元，用户B在8000元，总共1400元，而不会出现用户A扣除了200元，但用户B未增加的情况（该情况，用户A和B均为600元，总共120元）。
* 隔离性（Isolation）：数据库允许多个并发事务同时对其数据进行读写和修改的能力，隔离性可以防止多个事务并发执行时由于交叉执行而导致数据的不一致，因为多个事务同时使用相同的数据时，不会相互干扰，每个事务都有一个完整的数据空间，对其他并发事务是隔离的。也就是说，消费者购买商品这个事务，是不影响其他消费者购买的。
* 持久性（Durability）：事务处理结束后，对数据的修改就是永久的，即便系统故障也不会丢失。

**隔离性**那边还有**四种隔离级别**比较重要：

- Read Uncommitted（读未提交），隔离性最差。意思是：**别人还没提交的数据你都能看到**。会导致**脏读（Dirty Read）**

举例

A：

```
UPDATE money = 500
-- 没提交
```

B：

```
SELECT → 500
```

结果：

A：

```
ROLLBACK;
```

实际上钱根本没变。

B：

```
读了假的数据
```

这就叫：**脏读**



- Read Committed（读已提交），Oracle 默认级别。意思是**只能看到别人已经提交的数据**。解决了**脏读**。但是会导致：**不可重复读（Non-repeatable Read）**

举例

B：

```
SELECT money → 1000
```

A：

```
UPDATE → COMMIT
```

B 再查一次：

```
SELECT money → 500
```

同一个事务里：

```
同一条数据两次读取结果不同
```

这就是：不可重复读

- Repeatable Read（可重复读），是**MySQL InnoDB 默认级别**。意思是同一个事务里，同一条记录多次读取结果必须一致。解决了：脏读和不可重复读。但是理论上会有**幻读（Phantom Read）**

举例

B：

```
SELECT
WHERE age > 20
→ 5 条数据
```

A：

```
INSERT 一条 age=25
COMMIT
```

B 再查：

```
→ 6 条数据
```

像“凭空多了一行”

这叫：幻读

- Serializable（串行化），隔离性最强。

意思：

```
所有事务排队执行
一个一个来
```

完全没有并发问题。

但是：

```
性能极差
```

基本不用。







### InnoDB如何解决幻读

**在一个事物场景下**，对于幻读现象，不建议将隔离级别升级为串行化，因为这会导致数据库并发性能很差。MySQL InnoDB 引擎的默认隔离级别虽然是「可重复读」，但是它很大程度上避免幻读现象。解决的方案有两种：  

- 针对快照读（普通 select 语句），是通过 MVCC 方式解决了幻读，因为可重复读隔离级别下，事务执行过程中看到的数据，一直跟这个事务启动时看到的数据是一致的，即使中途有其他事务插入了一条数据，是查询不出来这条数据的，所以就很好地避免了幻读问题。  


- 针对当前读（select ...for update 等语句），是通过 next-key lock（记录锁+间隙锁）方式解决了幻读，因为当执行 select ...for update 语句的时候，会加上 next-key lock（**直到事务结束（COMMIT / ROLLBACK）才释放**），如果有其他事务在 next-key lock 锁范围内插入了一条记录，那么这个插入语句就会被阻塞，无法成功插入，所以就很好地避免了幻读问题。





### 什么是MVCC

MVCC（Multi-Version Concurrency Control，多版本并发控制）是 InnoDB 为了解决读写并发冲突、提高并发性能而实现的一种机制。

它的核心思想不是通过加锁来解决读写冲突，而是通过维护数据的多个版本，让读操作去访问历史快照版本，从而实现“读不阻塞写，写也不阻塞读”。

具体来说：

InnoDB 在每表的行记录后面都会维护两个隐藏字段：

1、trx_id：记录最后一次修改该行的事务 ID  
2、roll_pointer：回滚指针，指向该行的上一个版本（由 undo log 保存，undo log 存储的更早的版本里面，每个版本也有自己的trx_id和roll_pointer）

多个历史版本通过 undo log 串成一个版本链。

当事务执行普通 SELECT 时（快照读），不会加锁，而是通过 Read View + 版本链 来判断当前事务能看到哪个版本的数据。

Read View 本质上是一个一致性视图，里面主要包含：

- m_ids：当前活跃事务 ID 列表
- min_trx_id：最小活跃事务 ID
- max_trx_id：下一个将被分配的事务 ID

通过将记录的 trx_id 与 Read View 进行比较，判断该版本是否对当前事务可见：

- 如果 trx_id < min_trx_id，说明事务早已提交，可见
- 如果 trx_id >= max_trx_id，说明事务在 Read View 创建后才开始，不可见
- 如果介于两者之间，则判断是否在 m_ids 中：
  - 在：说明事务仍未提交，不可见
  - 不在：说明已提交，可见

如果当前版本不可见，就沿着 undo log 通过 roll_pointer 找更早的版本，直到找到可见版本。

这样就避免了大量加锁，提高了数据库并发性能。

需要注意：

MVCC 主要作用于 Read Committed 和 Repeatable Read 隔离级别；
其中：

- RC：每次 SELECT 都生成新的 Read View
- RR：事务第一次 SELECT 时生成 Read View，整个事务期间复用同一个 Read View

这也是 RR 能实现可重复读的根本原因。

总结一句话：

MVCC 的本质就是：
通过 undo log 保存历史版本，通过 Read View 判断版本可见性，让读操作读取快照数据，而不是阻塞等待写操作完成。

延伸一下：

对于「读提交」和「可重复读」隔离级别的事务来说，它们是通过 Read View 来实现的，它们的区别在于创建 Read View 的时机不同：

- 「读提交」隔离级别是在每个 select 都会生成一个新的 Read View，也意味着，事务期间的多次读取同一条数据，前后两次读的数据可能会出现不一致，因为可能这期间另外一个事务修改了该记录，并提交了事务。
- 「可重复读」隔离级别是启动事务时生成一个 Read View，然后整个事务期间都在用这个 Read View，这样就保证了在事务期间读到的数据都是事务启动前的记录。

这两个隔离级别实现是通过「事务的 Read View 里的字段」和「记录中的两个隐藏列」的比对，来控制并发事务访问同一个记录时的行为，这就叫 MVCC（多版本并发控制）。

在可重复读隔离级别中，普通的 select 语句就是基于 MVCC 实现的快照读，也就是不会加锁的。而 select ... for update 语句就不是快照读了，而是当前读了，也就是每次读都是拿到最新版本的数据，但是它会对读到的记录加上 next-key lock 锁。







## Spring

### Spring 事务为什么会失效

Spring 事务本质上是基于 AOP 实现的代理机制，只有通过 Spring 创建的代理对象调用方法时，事务才会生效。

常见的事务失效场景主要有：

第一，类内部自调用，例如一个方法中通过 `this.xxx()` 调用另一个加了 `@Transactional` 的方法，由于没有经过代理对象，事务不会生效。

第二，事务方法不是 public，例如 private、final、static 方法，可能无法被 Spring 正常代理。

第三，异常被自己 try-catch 捕获后没有继续抛出，Spring 感知不到异常，就不会触发事务回滚。

第四，抛出的是 checked exception，例如 `Exception`，而不是 `RuntimeException`，默认情况下 Spring 不会回滚，需要通过 `rollbackFor` 显式指定。

第五，底层数据库引擎本身不支持事务，例如 MyISAM，即使写了 `@Transactional` 也无法真正回滚。

第六，当前对象不是由 Spring 容器管理，而是手动 new 出来的对象，也不会有事务代理。

因此事务是否生效，核心取决于是否真正走了 Spring 的代理机制。
